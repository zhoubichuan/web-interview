# 二、蘑菇街

## 1.HTML 语义化的理解

- 使用正确标签表达内容结构，比如 `header`、`nav`、`main`、`article`、`section`、`footer`。
- 好处是提升可读性、SEO、无障碍访问能力，也方便浏览器和搜索引擎理解页面结构。

## 2.b 和 strong 的区别

- `b` 只表示视觉上的加粗，没有语义强调。
- `strong` 表示内容重要，默认也是加粗，但会被搜索引擎和辅助设备识别为强调内容。

## 3.对闭包的理解

- 闭包是函数和其词法作用域的组合，内部函数可以访问外部函数已经执行完后的变量。
- 本质是作用域链仍然引用外层变量，所以这些变量不会被垃圾回收。
- 常见用途有封装私有变量、缓存状态、函数柯里化和防抖节流。

## 4.工程中闭包使用场景

- 防抖、节流中保存定时器。
- 模块化代码中隐藏私有变量，只暴露必要 API。
- 事件回调、异步回调中保留当前上下文数据。

## 5.介绍 this 和原型

- `this` 的值由调用方式决定：普通函数看调用者，箭头函数继承定义时外层作用域。
- 原型是对象属性查找的后备链，实例通过 `__proto__` 指向构造函数的 `prototype`。
- 原型链可以实现属性和方法复用，也是 JS 继承机制的基础。

## 6.使用原型最大的好处

- 多个实例可以共享同一份方法，减少重复创建函数带来的内存开销。
- 可以通过原型链实现继承和能力扩展。

## 7.react 设计思路

- 用组件拆分 UI，用状态驱动视图更新。
- 通过虚拟 DOM 描述 UI，再用 diff 找出最小变更提交到真实 DOM。
- 数据流倾向单向流动，便于定位状态来源和维护复杂页面。

## 8.为什么虚拟 DOM 比真实 DOM 性能好

- 虚拟 DOM 操作的是 JS 对象，计算成本通常低于频繁操作真实 DOM。
- React 会批量更新，并通过 diff 减少真实 DOM 修改次数。
- 严格来说虚拟 DOM 不是永远更快，它主要提升复杂 UI 下的可维护性和更新性能下限。

## 9.react 常见的通信方式

- 父传子使用 `props`，子传父通过回调函数。
- 兄弟组件可通过共同父组件提升状态。
- 跨层级通信可用 `Context`，复杂全局状态可用 Redux、MobX 等状态管理方案。

## 10.redux 整体的工作流程

- 组件通过 `dispatch(action)` 描述发生了什么。
- `reducer` 根据旧 `state` 和 `action` 返回新 `state`。
- `store` 保存状态并通知订阅者，React 组件收到新状态后重新渲染。

## 11.redux 和全局对象之间的区别

- Redux 的状态变化必须通过 `dispatch` 和 `reducer`，可预测、可追踪。
- 全局对象可以被任意代码直接修改，来源不清晰，难以调试。
- Redux 配合中间件和 DevTools 可以记录、回放和调试状态变更。

## 12.Redux 数据回溯设计思路

- Redux 每次更新都返回新的不可变状态，不直接修改旧状态。
- DevTools 记录每次 `action` 和对应状态快照。
- 回溯时按 action 序列重新计算或切换快照，从而实现 time travel。

## 13.单例、工厂、观察者项目中实际场景

- 单例：全局状态管理、弹窗管理器、日志实例。
- 工厂：根据不同类型创建组件、请求实例或策略对象。
- 观察者：事件监听、数据订阅、响应式系统、发布订阅中心。

## 14.项目中树的使用场景以及了解

- 常见场景有 DOM 树、路由树、菜单树、组织架构树、文件目录树。
- 常用操作包括 DFS、BFS、递归渲染、查找节点、过滤节点和树转列表。
