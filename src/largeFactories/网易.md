# 十四、网易

## 1.介绍 redux，主要解决什么问题

- Redux 是集中式状态管理方案，通过单一 store 管理应用状态。
- 它主要解决跨组件状态共享、状态变化不可追踪、复杂页面数据流混乱的问题。
- 核心流程是 `dispatch(action)` -> `reducer` -> `new state` -> 视图更新。

## 2.文件上传如何做断点续传

- 前端把文件按固定大小切片，每片计算 hash 或携带索引上传。
- 服务端记录已上传分片，失败后前端查询已上传列表，只补传缺失分片。
- 所有分片上传完成后，服务端按顺序合并文件，并校验完整性。

## 3.表单可以跨域吗

- 原生表单提交可以跨域发起请求，但浏览器不会允许前端 JS 直接读取跨域响应内容。
- 如果需要读取响应，仍然需要 CORS、JSONP、代理等跨域方案。

## 4.promise、async 有什么区别

- `async/await` 是基于 Promise 的语法糖，让异步代码写起来更像同步流程。
- Promise 更适合组合链式调用；`async/await` 更适合顺序流程和 `try/catch` 错误处理。
- `async` 函数总是返回 Promise，`await` 后面的代码会进入微任务继续执行。

## 5.搜索请求如何处理（防抖）

- 输入框搜索通常使用防抖，用户停止输入一段时间后再发请求。
- 还需要处理竞态问题：可取消旧请求，或用请求序号保证只使用最后一次响应。

## 6.搜索请求中文如何请求

- 中文参数需要使用 `encodeURIComponent` 编码后放入 URL。
- 服务端按统一字符集解码，通常使用 UTF-8。
- 如果是 `POST`，也可以放在 JSON body 中传递，减少 URL 编码问题。

## 7.介绍观察者模式

- 观察者模式是一对多依赖关系，被观察对象状态变化时通知所有观察者。
- 典型场景包括事件监听、响应式数据更新、状态订阅。

## 8.介绍中介者模式

- 中介者模式用一个中心对象协调多个对象之间的交互。
- 各对象不直接互相依赖，而是通过中介者通信，降低耦合。
- 常见场景有事件总线、表单联动、模块通信中心。

## 9.观察者和订阅-发布的区别，各自用在哪里

- 观察者模式中目标对象直接维护观察者列表，状态变化时直接通知。
- 发布-订阅模式中有事件中心，发布者和订阅者通过 topic 解耦。
- 观察者常用于响应式系统，发布订阅常用于跨模块事件通信。

## 10.介绍 react 优化

- 使用 `React.memo`、`PureComponent`、`useMemo`、`useCallback` 减少重复渲染。
- 列表使用稳定 `key`，长列表使用虚拟滚动。
- 合理拆分状态和组件，避免父级更新导致大面积子组件更新。

## 11.介绍 http2.0

- HTTP/2 使用二进制分帧，支持多路复用，多个请求可以复用同一个 TCP 连接。
- 支持头部压缩 HPACK，减少重复 header 体积。
- 还支持请求优先级和服务端推送，但服务端推送在实际应用中使用较少。

## 12.通过什么做到并发请求

- 浏览器可以同时发起多个异步请求，如 `Promise.all`、并行 `fetch`。
- HTTP/1.1 下同域名连接数有限；HTTP/2 通过多路复用在单连接上并发多个 stream。
- 业务上可用队列控制并发数量，避免瞬间请求过多。

## 13.http1.1 时如何复用 tcp 连接

- HTTP/1.1 默认开启持久连接 `Connection: keep-alive`。
- 同一 TCP 连接可以串行复用多个请求，减少重复握手成本。
- 但同一连接内仍可能有队头阻塞问题。

## 14.介绍 service worker

- Service Worker 是运行在浏览器后台的脚本，独立于页面生命周期。
- 可以拦截网络请求，实现离线缓存、资源预缓存、消息推送等能力。
- 只能运行在 HTTPS 或 localhost 环境，并且生命周期由浏览器管理。

## 15.介绍 css3 中 position:sticky

- `position: sticky` 是相对定位和固定定位的结合。
- 元素在滚动到指定阈值前按普通文档流布局，超过阈值后固定在指定位置。
- 需要设置 `top`、`bottom` 等阈值，且受父级滚动容器和 `overflow` 影响。

## 16.redux 请求中间件如何处理并发

- Redux 中间件可以拦截 action，发起异步请求后再 dispatch 成功或失败 action。
- 并发请求可用请求 id、时间戳或取消令牌区分，保证只处理最新响应。
- 常见方案有 `redux-thunk`、`redux-saga`、`redux-observable`。

## 17.介绍 Promise，异常捕获

- Promise 有 `pending`、`fulfilled`、`rejected` 三种状态，状态一旦改变不可逆。
- 构造函数同步执行，`then/catch/finally` 回调进入微任务队列。
- 异常可通过 `catch` 捕获；链式调用中抛出的错误会传递到后续最近的 `catch`。

## 18.介绍 position 属性包括 CSS3 新增

- `static` 默认定位，不参与偏移。
- `relative` 相对自身原位置偏移，仍占据原空间。
- `absolute` 相对最近的定位祖先定位，脱离文档流。
- `fixed` 相对视口固定，`sticky` 根据滚动位置在相对和固定之间切换。

## 19.浏览器事件流向

- 捕获阶段 addeventListener true
- 目标阶段
- 冒泡阶段 addeventListener false
  - 阻止冒泡 e.stopPropagation
  - 阻止默认行为 e.preventDefault

## 20.介绍事件代理以及优缺点

- 事件代理是把子元素事件统一绑定到父元素上，利用事件冒泡判断触发源。
- 优点是减少事件绑定数量，动态新增子元素也能响应。
- 缺点是依赖冒泡，不适合不冒泡事件；逻辑过多时父级处理函数会变复杂。

## 21.React 组件中怎么做事件代理

- React 中直接在父组件上绑定事件，通过 `event.target` 或自定义属性判断具体子元素。
- 列表场景可以把 id 放在闭包、`data-*` 属性或回调参数里。

## 22.React 组件事件代理的原理

- React 会把大多数事件委托到根容器上统一监听。
- 原生事件触发后，React 包装成合成事件并按组件树路径派发。
- 这样可以屏蔽浏览器差异，也减少大量 DOM 节点上的事件绑定。

## 23.介绍 this 各种情况

- 函数作为对象函数被调用时,this 指向调用者
- 函数作为普通函数调用时，this 指向 window
- 函数作为构造函数时，this 指向构造函数
- this 的隐性丢失
- 箭头函数，this 指向上一层的 this

## 24.前端怎么控制管理路由

- SPA 中通常使用前端路由库维护 URL 和组件映射关系。
- `hash` 模式依赖 `hashchange`，`history` 模式依赖 `pushState/replaceState` 和 `popstate`。
- 路由层还会管理权限、懒加载、滚动恢复和页面状态。

## 25.使用路由时出现问题如何解决

- `history` 模式刷新 404，需要服务端配置 fallback 到入口 HTML。
- 路由权限问题可用全局守卫或高阶组件拦截。
- 参数变化组件不刷新时，可监听路由参数变化或给组件设置合适的 `key`。

## 26.React 怎么做数据的检查和变化

- 类型检查可用 TypeScript 或 `PropTypes`。
- 数据变化通常通过 `setState`、`useState`、`useReducer` 或外部状态管理触发。
- React 根据状态变化重新渲染组件，再通过 diff 更新真实 DOM。
